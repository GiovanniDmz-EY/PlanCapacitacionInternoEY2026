1. Fundamento universal de una aplicación

Interfaz

  • Medio de interacción.
  • Gráfica (web/móvil), CLI (Command-Line Interface, interfaz por línea de comandos) o API (Application Programming Interface, interfaz para que sistemas se comuniquen entre sí).

Lógica / procesamiento

  • Recepción, validación y transformación de datos.
  • Reglas de negocio y generación de resultados.

Datos

  • Temporales (memoria) y persistentes.
  • Estructurados, no estructurados, semi-estructurados y vectoriales.

2. Evolución de las aplicaciones y variaciones del mercado

  • Tradicional: monolíticas en un solo sistema.
  • Transición web: separación frontend, backend y base de datos.
  • Variaciones del mercado: interfaces web/mobile/multiplataforma, múltiples lenguajes y frameworks, integración por APIs y consumo humano/sistema y sistema/sistema.
  • Resultado: arquitecturas desacopladas, backend único y responsabilidades separadas.

3. Problemas al desacoplar sistemas

  • Comunicación entre componentes.
  • Seguridad e identidad.
  • Monitoreo, costos y gobierno.

4. Modelos de servicio y ejecución

  • IaaS (Infrastructure as a Service): control de infraestructura y del sistema operativo.
  • PaaS (Platform as a Service): infraestructura administrada y foco en la aplicación.
  • SaaS (Software as a Service): software completamente administrado.
  • Serverless: ejecución bajo demanda, sin gestión de servidores y basada en eventos.

5. Consideraciones para la creación de recursos

  • Ubicación (región): afecta latencia, costo, disponibilidad y cumplimiento legal (soberanía de datos).
  • Capacidad / SKU: define nivel de servicio, límites de hardware y recursos asignados. SKU es la combinación comercial/técnica que determina tamaño, desempeño y precio.
  • Modelo de ejecución: define si el cobro es por recurso siempre activo, serverless o escalado automático.
  • Acceso y seguridad: define visibilidad pública/privada, autenticación y restricciones de red (IPs).
  • Integraciones / dependencias: relación con otros servicios y recursos implícitos (discos, NICs, IPs).
  • Impacto de configuración: decisiones de identidad (nombre) y ubicación difíciles de modificar sin recrear.
  • Jerarquía y Organización: uso de Suscripciones y Resource Groups para agrupación lógica y de costos.
  • Tags: clasificación por ambiente, proyecto y responsable para auditoría y facturación.
  • Gobernanza y Bloqueos: uso de Resource Locks (bloqueos a nivel de recurso) para evitar borrado accidental o modificaciones críticas.

Geografías, regiones y zonas

  • Geografía: agrupación amplia de regiones, por ejemplo Europa o América.
  • Región: ubicación específica de Azure donde se despliegan recursos, como East US o West Europe.
  • Zona de disponibilidad: datacenter físicamente separado dentro de una misma región, usado para mejorar resiliencia local.
  • Par de regiones: relación predefinida entre dos regiones dentro de una geografía para apoyar estrategias de continuidad y mantenimiento.

Azure Resource Manager

  • Azure Resource Manager (ARM): capa de administración y despliegue de recursos en Azure.
  • Es el modelo que entiende recursos, grupos de recursos, dependencias, plantillas y permisos.
  • Cuando despliegas con portal, plantillas ARM, Bicep, CLI o PowerShell, normalmente estás operando sobre ARM.

CLI y PowerShell

  • Azure CLI: herramienta por línea de comandos multiplataforma para administrar Azure de forma declarativa o scriptable.
  • Azure PowerShell: conjunto de módulos PowerShell para administrar Azure desde scripts y automatizaciones orientadas a ese ecosistema.
  • Ambas permiten operar recursos sin depender del portal y son muy usadas en automatización y administración.

6.1 Azure Virtual Machine

  1. Definición y modelo de servicio
    • Concepto: instancia de cómputo que emula un servidor físico con CPU, memoria y almacenamiento.
    • Modelo: corresponde a IaaS, por lo que el servidor es visible y el usuario administra sistema operativo, parches, middleware y aplicaciones.
    • Responsabilidad del proveedor: Microsoft gestiona el hardware físico, la disponibilidad base y la capa de virtualización.
  2. Componentes y dependencias
    • Network Interface (NIC): tarjeta o interfaz de red virtual que conecta la VM a la red y le da identidad de comunicación.
    • Virtual Network (VNet) y Subnet: segmento lógico donde vive la máquina y desde donde obtiene direccionamiento interno.
    • Managed Disks: almacenamiento persistente para sistema operativo y datos.
    • Public IP: componente opcional cuando se requiere exposición o administración desde Internet.
  3. Modelo de ejecución y operación
    • Ejecución continua: la capacidad permanece asignada mientras la VM está encendida.
    • Operación directa: el equipo administra arranque, apagado, mantenimiento del sistema y actualizaciones.
    • Boot Diagnostics: ayuda a diagnosticar errores de inicio del sistema operativo cuando no hay acceso por RDP (Remote Desktop Protocol, acceso remoto gráfico a Windows) o SSH (Secure Shell, acceso remoto seguro por terminal a Linux).
  4. Conectividad, acceso y seguridad
    • Conectividad interna: puede usar IPs privadas para comunicarse con otros servicios de Azure sin exponer tráfico a Internet.
    • Conectividad híbrida: puede integrarse con redes on-premise mediante gateways o VPNs.
    • Protocolos de acceso: RDP para Windows y SSH para Linux.
    • Autenticación: puede basarse en usuario y contraseña, claves SSH o integración con Entra ID, según el sistema operativo.
    • Seguridad recomendada: evitar acceso público directo cuando no es necesario y preferir VPN o Azure Bastion.
    Idea clave: una VM no solo se conecta hacia Internet; también forma parte de una red interna. Diseñar bien la conectividad evita exponer recursos innecesariamente.
  5. Escalabilidad
    • Escalado vertical: cambiar el SKU para aumentar o disminuir CPU y memoria; normalmente requiere reinicio.
    • Escalado horizontal: usar Virtual Machine Scale Sets (VMSS, grupo administrado de múltiples VMs) para desplegar varias instancias detrás de un balanceador.
    • Objetivo: absorber picos de demanda distribuyendo la carga entre múltiples nodos.
  6. Costos y eficiencia operativa
    • Costo principal: se cobra por capacidad provisionada mientras la VM está activa.
    • Auto-shutdown: ayuda a reducir costos en entornos no productivos apagando la instancia automáticamente.
    • Estado Deallocated: elimina el costo de cómputo, pero pueden mantenerse cargos por discos e IPs reservadas.
    • Spot Instances: aprovechan capacidad excedente con descuento, pero pueden ser desalojadas con muy poca anticipación.
    Nota de arquitectura: Spot Instances solo debe usarse en escenarios tolerantes a interrupciones, como pruebas, desarrollo o procesos no críticos.
  7. Casos de uso y consideraciones de arquitectura
    • Casos de uso: migraciones lift-and-shift, aplicaciones legacy, software con dependencias específicas o necesidad de control total del entorno.
    • Persistencia: los discos administrados sobreviven a la instancia y deben gestionarse para evitar recursos huérfanos.
    • Delete with VM: configuración recomendada para limpiar automáticamente discos e interfaces al eliminar la instancia.
    • Consideración: ofrece máximo control, pero también la mayor responsabilidad operativa de las tres opciones.
    Síntesis: Azure VM es la opción adecuada cuando se necesita control detallado del servidor, aceptando que el equipo también asume más tareas de operación y seguridad.

6.2 Azure App Service

  1. Definición y modelo de servicio
    • Concepto: servicio PaaS para desplegar aplicaciones web, APIs y backends sin administrar directamente sistema operativo, parches o servidor.
    • Diferencia clave frente a VM: la infraestructura sigue existiendo, pero se abstrae para que el equipo se concentre en la aplicación y su configuración.
    • Separación importante: la aplicación no es la infraestructura; el recurso visible para el usuario y la capacidad de cómputo son componentes distintos.
  2. Componentes y dependencias
    • App Service: representa la aplicación publicada, su configuración y su punto de exposición.
    • App Service Plan: define la infraestructura real donde corre la aplicación: CPU, memoria, escalabilidad y región.
    • Modelo compartido: varias aplicaciones pueden hospedarse dentro del mismo plan y comparten la capacidad disponible.
    • Implicación operativa: el costo y la potencia no dependen directamente del número de apps, sino del plan que las sostiene.
    Idea clave: App Service es la aplicación; App Service Plan es el servidor abstracto que realmente aporta la capacidad de ejecución.
  3. Modelo de ejecución y operación
    • Ejecución continua: los recursos del plan permanecen asignados según la capacidad configurada.
    • Code-based: se despliega el código y la plataforma administra el runtime compatible.
    • Container-based: se despliega una imagen de contenedor, dando mayor control sobre dependencias y entorno de ejecución.
    • Deployment slots: entornos paralelos dentro del mismo App Service, por ejemplo `staging` y `production`.
    • Swap: una versión validada puede intercambiarse con producción sin redeploy completo, reduciendo riesgo e interrupciones.
  4. Conectividad, acceso y seguridad
    • Exposición por defecto: la aplicación suele publicarse mediante URL HTTP/HTTPS accesible desde Internet.
    • Configuración desacoplada: variables de entorno y cadenas de conexión separadas del código.
    • Manejo de secretos: puede integrarse con servicios especializados para evitar credenciales embebidas en la aplicación.
    • Controles de acceso: restricciones por IP, reglas de acceso y configuración CORS según el tipo de consumidor.
  5. Escalabilidad
    • Escalado vertical: aumentar la capacidad del plan, por ejemplo más CPU o memoria.
    • Escalado horizontal: incrementar el número de instancias para distribuir carga.
    • Impacto compartido: si varias aplicaciones usan el mismo plan, el escalado afecta a todas porque comparten la misma base de infraestructura.
  6. Costos y eficiencia operativa
    • Costo principal: depende del App Service Plan y de la capacidad provisionada, no del número de aplicaciones.
    • Recurso siempre activo: aunque la aplicación no reciba tráfico, la infraestructura del plan sigue generando costo.
    • Eficiencia: varias aplicaciones pueden compartir el mismo plan para optimizar costo, siempre que la capacidad esté bien dimensionada.
  7. Casos de uso y consideraciones de arquitectura
    • Uso recomendado: aplicaciones web, APIs y backends que requieren disponibilidad continua sin administrar infraestructura base.
    • No ideal para: cargas puramente event-driven o escenarios donde se requiere control total del sistema operativo.
    • Dependencias complementarias: monitoreo, configuración de red, almacenamiento y otros servicios de soporte pueden formar parte de la solución.
    • Consideración: es un punto intermedio entre una VM y un modelo serverless, con menor complejidad operativa pero también menos control del entorno.
    Síntesis: App Service es un punto intermedio entre una VM y un modelo serverless. Simplifica la operación, pero exige entender bien el App Service Plan para diseñar capacidad, costos y aislamiento.

6.3 Azure Container Apps

  1. Definición y modelo de servicio
    • Concepto: servicio PaaS orientado a ejecutar aplicaciones basadas en contenedores sin administrar directamente servidores ni orquestadores.
    • Modelo: abstrae la infraestructura base, pero conserva conceptos propios del mundo de contenedores como imagen, revisión, ingress y escalado por eventos.
    • Unidad de despliegue: la aplicación se publica a partir de una imagen de contenedor que incluye runtime, dependencias y configuración.
  2. Componentes y dependencias
    • Container Registry: almacena y versiona las imágenes de contenedores; no ejecuta aplicaciones.
    • Container App: es la aplicación en ejecución que toma una imagen y la corre.
    • Container Apps Environment: es el entorno compartido donde viven las aplicaciones y donde se define red, aislamiento y configuración compartida.
    • Integración: el acceso al registry puede resolverse con credenciales o identidad administrada.
  3. Modelo de ejecución y operación
    • Ejecución flexible: permite escalado automático e incluso escalar a cero cuando no hay carga.
    • Flujo típico: construir la imagen, publicarla en un registry y desplegar una nueva revisión en la Container App.
    • Revisions: permiten versionar despliegues y controlar la transición entre versiones.
  4. Conectividad, acceso y seguridad
    • Ingress: define cómo se expone la aplicación y cómo recibe tráfico.
    • Exposición: puede configurarse como pública o interna.
    • Container Apps Environment: también define comunicación entre servicios e integración con red privada (VNet).
    • Configuración: admite variables de entorno, secrets y parámetros por aplicación.
  5. Escalabilidad
    • Escalado por HTTP: responde al volumen de tráfico web.
    • Escalado por eventos: puede reaccionar a colas, mensajes u otros disparadores.
    • Límites configurables: se define mínimo y máximo de instancias según la carga esperada.
  6. Costos y eficiencia operativa
    • Factores de costo: CPU, memoria, tiempo de ejecución y número de instancias.
    • Servicios adicionales: logs, registry y red pueden incrementar el costo total de la solución.
    • Eficiencia: la capacidad de escalar a cero puede volverlo más eficiente que un servicio siempre activo en cargas variables.
  7. Casos de uso y consideraciones de arquitectura
    • Casos de uso: microservicios, APIs en contenedores, aplicaciones event-driven y cargas variables.
    • Observabilidad: incluye logs, métricas y monitoreo para analizar ejecución, errores y consumo de recursos.
    • Consideración: ofrece mayor flexibilidad que App Service para contenedores, pero introduce conceptos adicionales como environments, revisions y escalado por eventos.
    Síntesis: Container Apps combina simplicidad PaaS con patrones modernos de contenedores, siendo especialmente útil cuando se necesita elasticidad y despliegue basado en imágenes.

6.4 Azure Kubernetes Service (AKS)

  1. Definición y modelo de servicio
    • Concepto: AKS es el servicio administrado de Kubernetes en Azure para desplegar y administrar aplicaciones en contenedores.
    • Qué administra Azure: Azure crea y opera el plano de control, que es la capa que coordina el clúster, programa cargas y mantiene el estado de Kubernetes.
    • Qué administra el cliente: el equipo sigue administrando las cargas de trabajo, la configuración del clúster, la seguridad operativa y los nodos de trabajo que ejecutan las aplicaciones.
    • Lectura práctica: AKS descarga mucha complejidad operativa de Kubernetes, pero no elimina la responsabilidad sobre diseño, seguridad, despliegues, costos y operación de las aplicaciones.
  2. Conceptos base: contenedor, pod, nodo y clúster
    • Contenedor: paquete ejecutable con aplicación, dependencias y runtime.
    • Pod: unidad básica de ejecución en Kubernetes; normalmente contiene uno o varios contenedores relacionados que comparten red y almacenamiento cercano.
    • Nodo: máquina virtual del clúster donde se ejecutan los pods; también se le suele llamar worker node porque es donde realmente corre el trabajo.
    • Clúster: conjunto completo de plano de control, nodos y componentes para ejecutar aplicaciones distribuidas.
    • Node pool: grupo de nodos con características similares dentro del mismo clúster, útil para separar tipos de carga o sistemas operativos.
    Forma simple de recordarlo: el contenedor empaqueta la app, el pod la ejecuta, el nodo hospeda pods y el clúster coordina todo el conjunto.
  3. Cómo funciona AKS y qué resuelve Kubernetes
    • Cuando creas un clúster, Azure aprovisiona y mantiene el plano de control de Kubernetes.
    • Luego despliegas uno o varios grupos de nodos para ejecutar las aplicaciones.
    • Kubernetes programa los pods en los nodos disponibles según recursos, reglas y estado general del clúster.
    • En el modelo estándar, Azure no cobra por separado el plano de control; el costo principal viene de los nodos y servicios asociados.
    • Service discovery: Kubernetes permite que los servicios se encuentren entre sí mediante DNS o IPs internas.
    • Load balancing: puede distribuir tráfico entre varias réplicas de una misma aplicación.
    • Self-healing: si un pod falla, Kubernetes intenta reiniciarlo o reprogramarlo para mantener el estado deseado.
    • Storage orchestration: permite montar almacenamiento persistente para que las aplicaciones no pierdan datos aunque el pod se reprograme.
    Lectura práctica del flujo: el usuario define el estado deseado, Kubernetes decide dónde y cómo ejecutar la carga, y la plataforma intenta mantenerla disponible automáticamente. Esa automatización y resiliencia es una de las razones principales para usar AKS.
  4. Componentes lógicos importantes
    • kube-apiserver: punto de entrada para la API de Kubernetes.
    • Scheduler: decide en qué nodo se ejecutará cada pod nuevo.
    • Controller Manager: vigila que el estado real del clúster coincida con el estado deseado.
    • kubelet: agente que corre en cada nodo y confirma que los contenedores realmente están ejecutándose.
    • Container runtime: componente del nodo que ejecuta los contenedores, por ejemplo containerd.
    Idea clave: no necesitas operar manualmente todos esos componentes en AKS, pero sí conviene saber qué papel juegan para entender incidentes, escalado y diseño del clúster.
  5. Red, acceso y seguridad
    • Integración de red: el clúster se conecta a una VNet y las aplicaciones suelen publicarse mediante servicios, ingress o balanceadores.
    • Identidad y acceso: puede integrarse con Microsoft Entra ID y RBAC para controlar administración y acceso al clúster.
    • Secretos y configuración: puede trabajar con secretos nativos de Kubernetes, Key Vault e identidades administradas.
    • Imagenes: normalmente consume imágenes desde un registry como Azure Container Registry (ACR).
  6. Escalabilidad, upgrades y operación
    • Escalado de nodos: el clúster puede aumentar o disminuir nodos según la demanda.
    • Escalado de pods: Kubernetes puede aumentar o disminuir réplicas de una aplicación sin cambiar necesariamente la cantidad de nodos.
    • Recuperación automática: si un pod falla, Kubernetes intenta reprogramarlo para mantener el estado deseado.
    • Upgrades: AKS ayuda a actualizar la versión del clúster y a mover cargas a nodos nuevos con menor fricción que un Kubernetes autoadministrado.
    • Node pools especializados: puedes tener pools para Linux, Windows, GPU o cargas con características distintas.
  7. Observabilidad e integración con Azure
    • Monitoreo: puede integrarse con Azure Monitor y Log Analytics para recolectar métricas y logs de nodos, clúster y contenedores.
    • Diagnóstico: la observabilidad es clave porque los problemas pueden venir de aplicaciones, red, nodos, scheduling o almacenamiento.
    • Ecosistema: AKS se integra bien con ACR, Entra ID, Policy, networking de Azure y pipelines DevOps.
  8. Costos y trade-offs
    • Costo principal: se paga por nodos, almacenamiento, red y servicios complementarios; el valor del servicio es que Azure administra el plano de control.
    • Complejidad: AKS ofrece mucha flexibilidad, pero requiere más conocimiento operativo que App Service, Functions o Container Apps.
    • Criterio de elección: conviene cuando realmente necesitas Kubernetes como plataforma de orquestación, no solo porque una app usa contenedores.
  9. Casos de uso y cuándo sí conviene
    • Microservicios: aplicaciones compuestas por varios servicios que necesitan despliegue independiente, comunicación interna y escalado coordinado.
    • Migración a contenedores: lift-and-shift de aplicaciones hacia un entorno administrado de Kubernetes.
    • Cargas avanzadas: escenarios con varios node pools, necesidades especiales de red, integración con DevOps y requerimientos de portabilidad.
    • No siempre es la primera opción: si el problema es más simple, App Service, Container Apps o Functions pueden resolver con menor complejidad.
    Síntesis: AKS es la opción adecuada cuando necesitas Kubernetes como plataforma de orquestación. Su valor no está solo en ejecutar contenedores, sino en coordinarlos, escalarlos y operarlos como una plataforma distribuida.

6.5 Azure Functions

  1. Definición y modelo de servicio
    • Categoría Cloud: clasificado oficialmente como PaaS (Platform as a Service).
    • Paradigma: Serverless / FaaS (Function as a Service).
    • Concepto: ejecución de lógica atómica bajo demanda. La plataforma abstrae el servidor, el parcheo y el mantenimiento para que el desarrollador se enfoque solo en el código.
  2. Componentes y dependencias
    • Function App: host lógico que agrupa las funciones y define el stack tecnológico como Python, .NET o Node.js.
    • Azure Storage Account (obligatoria): no es opcional. Se usa para persistir el código, gestionar las llaves de seguridad, los logs de ejecución y los puntos de control de eventos.
    • Triggers y Bindings: el trigger despierta la función; los bindings eliminan el código de infraestructura para leer o escribir en servicios como Storage, SQL o Cosmos DB de forma automática.
  3. Modelo de ejecución y operación
    • Event-driven: el código solo se ejecuta si un evento lo dispara.
    • Flex Consumption: es el estándar moderno para Functions y agrega control fino sobre concurrencia y memoria sin perder el modelo serverless.
    • Control de concurrencia: permite definir cuántas peticiones atiende cada instancia antes de que Azure levante otra, evitando saturar recursos como bases de datos.
    • Memoria seleccionable: permite asignar 2,048MB o 4,096MB de RAM por instancia para procesos más pesados, como procesamiento de imágenes o datos en Python.
    • Planes alternativos: Consumption para pago por ejecución puro, Premium para mantener instancias siempre activas y Dedicated cuando se ejecuta sobre un App Service Plan.
  4. Conectividad, acceso y seguridad
    • VNet Injection: en Flex permite conectarse a recursos en redes privadas sin recurrir al costo elevado del plan Premium, manteniendo el modelo de pago por uso.
    • Managed Identities: autenticación automática con otros servicios de Azure sin necesidad de escribir contraseñas o connection strings en el código.
    • Authorization Keys: protección de endpoints HTTP mediante llaves de acceso de nivel función o host.
  5. Escalabilidad
    • Elasticidad dinámica: el escalado es automático y basado en la carga de eventos.
    • Escala a cero: si no hay eventos, no existe consumo de recursos ni cobro.
    • Velocidad sub-segundo: Flex puede instanciar múltiples copias de la función en menos de un segundo para responder a picos masivos de tráfico.
  6. Costos y eficiencia operativa
    • Modelo de pago: en Flex Consumption se factura por el tiempo de ejecución de las instancias y la memoria consumida.
    • Eficiencia: elimina el costo por capacidad ociosa; a diferencia de una VM o un App Service Plan, no se paga por un servidor esperando trabajo.
    • Optimización: al poder elegir cuánta RAM usa la función, se reducen errores de out of memory que antes obligaban a migrar a planes mucho más caros.
  7. Casos de uso y consideraciones de arquitectura
    • Escenarios ideales: procesamiento de archivos en tiempo real, webhooks, tareas programadas y APIs de microservicios que requieren conectividad privada.
    • Integración con contenedores: el modelo de Functions también puede ejecutarse dentro de Azure Container Apps, unificando funciones con microservicios.
    • Limitación: no es apto para procesos de larga duración que excedan los límites de timeout de la plataforma.
    Síntesis: Azure Functions en Flex Consumption resuelve el dilema entre red privada empresarial y economía serverless, ofreciendo conectividad segura, memoria ajustable y pago por uso en una infraestructura prácticamente invisible.

7.1 Azure Storage Account

  1. Paradigma fundamental y modelo de costos
    • Modelo: Azure Storage funciona bajo pago por uso; no existe una tarifa plana única.
    • Factores de costo: la factura combina capacidad almacenada, transacciones, tier de acceso, redundancia y transferencia de salida de red.
    • Ingress vs. egress: subir datos normalmente no genera cobro, pero descargar o mover datos fuera de la región sí puede hacerlo.
  2. Servicios centrales de una Storage Account
    • Blobs: almacenamiento de objetos binarios para datos no estructurados como imágenes, videos y respaldos.
    • Files: recursos compartidos SMB/NFS (protocolos de compartición de archivos en red) totalmente gestionados, útiles para aplicaciones legacy o montajes simultáneos en varias VMs o pods.
    • Queues: colas de mensajes para desacoplar componentes y absorber picos de carga.
    • Tables: almacén NoSQL de clave-valor para grandes volúmenes de datos estructurados sin modelo relacional.
    • Capacidad adicional: también puede habilitarse hosting web estático sobre blobs para publicar frontends simples.
  3. Access tiers y trade-off de hardware
    • Hot: mayor costo de almacenamiento y menor costo operativo; ideal para datos de acceso frecuente y baja latencia.
    • Cool: menor costo de almacenamiento y mayor costo de acceso; adecuado para datos esporádicos con disponibilidad inmediata.
    • Archive: costo de almacenamiento mínimo, pero no accesible directamente; requiere rehidratación para volver a Hot o Cool antes de leerse.
    • Retenciones mínimas: Cool penaliza por menos de 30 días y Archive por menos de 180 días.
  4. Resiliencia y protección de datos
    • Redundancia: LRS (Locally Redundant Storage) mantiene copias locales, ZRS (Zone-Redundant Storage) replica entre zonas de disponibilidad y GRS (Geo-Redundant Storage) agrega copia geográfica en otra región.
    • Soft delete: actúa como una papelera de reciclaje para blobs y files.
    • Blob versioning: conserva versiones anteriores de los archivos.
    • Inmutabilidad (WORM): bloquea modificación o eliminación durante un periodo definido; WORM significa Write Once, Read Many, es decir, escribir una vez y leer muchas.
    • Lifecycle Management: automatiza transiciones entre Hot, Cool, Archive o eliminación según antigüedad o reglas de negocio.
  5. Arquitectura enterprise y seguridad
    • Seguridad de red: por defecto puede ser pública; en producción conviene restringir acceso y usar Private Endpoints para exponerla dentro de una VNet con IP privada.
    • Acceso público en blobs: en Blob Storage el acceso anónimo se controla en dos niveles: la Storage Account puede permitir o bloquear acceso público a blobs, y cada container define si admite lectura anónima o permanece privado.
    • Jerarquía del permiso: si la cuenta bloquea acceso público, ningún contenedor ni blob puede exponerse anónimamente. Si el contenedor está en modo privado, entonces los blobs dentro de él tampoco pueden quedar públicos de forma individual; prevalece la restricción del contenedor.
    • Exposición controlada con SAS: cuando se necesita compartir temporalmente un archivo sin volver público el contenedor, se puede generar una URL con Shared Access Signature (SAS), un token temporal que delimita tiempo, permisos y alcance. En general conviene que sea de corta duración y lo más específica posible.
    • Seguridad de identidad: evitar Access Keys compartidas y preferir Microsoft Entra ID con RBAC y Managed Identities.
    • Gobernanza: Azure Policy ayuda a imponer configuraciones seguras, por ejemplo prohibiendo acceso público.
    • Big data analytics: con Hierarchical Namespace (jerarquía real de carpetas) la cuenta se comporta como ADLS Gen2 (Azure Data Lake Storage Gen2), habilitando estructuras de carpetas reales para escenarios con Databricks o Synapse.
    • Recuperación regional: la replicación asíncrona de objetos entre cuentas puede apoyar estrategias de disaster recovery.
    Síntesis: Una Storage Account no es solo un repositorio de archivos. Es una plataforma de datos con múltiples servicios, costos multidimensionales, niveles de acceso con trade-offs claros y controles de resiliencia, visibilidad y seguridad que deben diseñarse desde el inicio.

7.2 Bases de datos relacionales en Azure SQL

En soluciones empresariales, la decisión clave no es solo "qué motor usar", sino cuánta operación quiere absorber el equipo, qué nivel de elasticidad necesita la carga y cómo se protegerá el acceso a los datos.

Modelo IaaS

  • SQL Server en Azure VM: control total del sistema operativo y del motor.
  • Escenario: migraciones lift-and-shift, dependencias legacy o features no disponibles en PaaS.
  • Trade-off: el equipo administra parches, backups, endurecimiento, monitoreo y diseño de alta disponibilidad.

Modelo PaaS

  • Azure SQL Database / Managed Instance: Microsoft opera la capa de plataforma.
  • Escenario: aplicaciones modernas, nuevos desarrollos y servicios que priorizan agilidad operativa.
  • Trade-off: se pierde parte del control fino del host, pero se reduce mucho la carga operativa.

Lectura arquitectónica

  • Responsabilidad compartida: el costo real no es solo financiero; también incluye administración diaria.
  • IaaS: más flexibilidad, más operación.
  • PaaS: menos operación, mejor velocidad para entornos productivos estandarizados.
ModeloCuándo usarloCosto / operación
SQL Server en Azure VM (IaaS) Migraciones lift-and-shift, dependencias de SO, control total del motor y configuraciones especiales. Se paga VM, almacenamiento y licencias o Azure Hybrid Benefit; el equipo opera parches, backups y HA.
Azure SQL Database (PaaS) Aplicaciones modernas con foco en simplicidad, elasticidad y menor administración. Se paga por capacidad contratada o consumida; Microsoft administra la plataforma y la disponibilidad base.
Azure SQL Managed Instance (PaaS) Workloads que necesitan alta compatibilidad con funcionalidades de nivel instancia de SQL Server sin operar una VM completa. Mantiene el modelo administrado, pero con un costo y una huella de red mayores que Azure SQL Database.

Azure SQL Managed Instance: qué resuelve en la práctica

Azure SQL Managed Instance no debe leerse solo como "un SQL Server muy parecido". La idea correcta es que ofrece un modelo PaaS, pero con compatibilidad mucho más cercana a una instancia tradicional de SQL Server que Azure SQL Database. Esto es especialmente valioso cuando se migra una aplicación existente que depende de comportamientos de nivel instancia y no solo de una base aislada.

No confundir el criterio

  • Azure SQL Database sí soporta tablas, vistas, procedimientos almacenados, funciones y gran parte de T-SQL (el dialecto SQL de Microsoft SQL Server).
  • Por lo tanto, la presencia de views o stored procedures por sí sola no obliga a usar Managed Instance.
  • La diferencia fuerte aparece cuando la solución depende de capacidades más cercanas a una instancia completa.

Cuándo empieza a importar

  • Cuando hay dependencias entre varias bases de datos dentro de la misma instancia.
  • Cuando la operación depende de automatización interna con SQL Server Agent (motor de jobs programados dentro de SQL Server).
  • Cuando existen componentes legacy o integraciones que esperan semántica más cercana a SQL Server tradicional.

Lectura arquitectónica

  • Azure SQL Database está más orientado al patrón "una base como servicio".
  • Managed Instance está orientado al patrón "una instancia administrada con varias capacidades tradicionales".
  • Por eso suele ser un mejor destino de migración para cargas heredadas.

Punto fino: si la necesidad es solo tener procedimientos almacenados, vistas, funciones o lógica T-SQL dentro de una base, Azure SQL Database sigue siendo válido. El salto a Managed Instance normalmente se justifica cuando la aplicación depende de características de instancia, no solo de objetos de base de datos.

Ejemplos concretos para elegir Managed Instance

Jobs internos del motor

  • Un sistema que ejecuta respaldos lógicos, mantenimiento o procesos nocturnos con SQL Server Agent Jobs.
  • Si esa operación ya vive dentro del motor y se quiere migrar con el menor rediseño posible, Managed Instance encaja mejor.

Dependencias entre bases

  • Aplicaciones heredadas que hacen consultas entre varias bases con referencias cruzadas o transacciones compartidas.
  • Ese patrón suele alinearse mejor con Managed Instance que con Azure SQL Database.

Migración con pocos cambios

  • Cuando el objetivo no es rediseñar la aplicación, sino moverla a PaaS manteniendo alto grado de compatibilidad.
  • Managed Instance reduce fricción frente a una reingeniería completa hacia una base más desacoplada.

Servidor lógico vs. base de datos

En la oferta PaaS de Azure SQL conviene separar dos recursos que suelen confundirse al inicio: por un lado está Azure SQL Server, que representa el servidor lógico; por otro, Azure SQL Database, que es la base de datos en sí. No cumplen el mismo papel y no se administran igual.

Azure SQL Server

  • No es una máquina virtual ni un SQL Server "instalado".
  • Es el servidor lógico donde se definen reglas de firewall, redes, autenticación con Entra ID y administración de usuarios a nivel de plataforma.
  • Sirve como punto común para varias bases de datos dentro del mismo contexto administrativo.

Azure SQL Database

  • Es el recurso individual donde viven las tablas, índices, vistas, procedimientos y datos.
  • Su rendimiento, almacenamiento y costo se definen a nivel de la base.
  • Un mismo servidor lógico puede hospedar varias bases de datos independientes.

Error común

  • Confundir Azure SQL Server con Azure SQL Database lleva a creer que el servidor lógico es una máquina o que la base incluye toda la configuración de red y acceso.
  • En PaaS, el punto de administración y el dato viven en capas distintas: primero se define el marco de acceso y luego se hospeda la base.

Dimensionamiento y modelo de compra

En costos y volumetría hay dos mecanismos principales: el modelo basado en DTU y el modelo basado en vCore. El más importante para diseño moderno es vCore, porque permite elegir de forma más explícita la capacidad, especialmente CPU y memoria. Dentro de vCore existen dos formas de oferta: aprovisionado y serverless.

Modelo DTU

  • Empaqueta CPU, memoria y IO en una unidad abstracta de compra.
  • Es más simple de consumir, pero menos transparente para dimensionar cargas complejas.
  • Suele verse como un modelo básico o heredado frente a vCore.

Modelo vCore

  • Permite elegir con mayor claridad la capacidad que necesita la base, especialmente CPU y memoria.
  • Da mejor control para alinear costo, rendimiento y tipo de carga.
  • Es el enfoque recomendado cuando se busca diseño más fino para producción.

vCore: aprovisionado vs. serverless

  • Aprovisionado: mantienes una capacidad seleccionada 24/7, incluso cuando no hay uso.
  • Serverless: ajusta procesamiento bajo demanda y puede aprovechar auto-pausa en periodos de inactividad.
  • Punto fino: si la base está en pausa, el primer usuario percibirá una latencia mientras el servicio "despierta".
Modelo de compraVentajaLimitación o advertencia
vCore aprovisionado Capacidad fija 24/7, latencia estable y control fino del dimensionamiento. Se sigue pagando la capacidad aunque la carga esté ociosa.
vCore serverless Elasticidad y ahorro cuando la carga es intermitente; con auto-pausa, en la práctica puede quedar cobrando casi solo almacenamiento. Hay que cuidar la auto-pausa: el primer acceso tras inactividad tendrá latencia mientras el servicio se reactiva.
DTU Modelo simple para entornos básicos o legados. Combina CPU, memoria e IO en un paquete menos transparente y menos flexible que vCore.

Niveles de servicio en vCore: General Purpose, Business Critical e Hyperscale

Cuando trabajas con el modelo vCore en Azure SQL Database, aparece otra decisión importante: el nivel de servicio. Este define el tipo de almacenamiento, el comportamiento de alta disponibilidad, la latencia esperada y el perfil de la carga. No debe confundirse con el modelo DTU, porque en DTU la clasificación tradicional es Basic, Standard y Premium.

La diferencia arquitectónica más importante entre General Purpose y Business Critical es esta: en General Purpose, el proceso de la base de datos y los archivos de datos/log viven en capas separadas; en Business Critical, el proceso y el almacenamiento activo están juntos en los nodos del clúster. En términos simples: en General Purpose el motor "va a buscar" los datos a una capa de almacenamiento remota; en Business Critical el motor los tiene mucho más cerca, sobre discos SSD locales, y por eso responde mejor en cargas transaccionales intensivas.

General Purpose

  • Es el nivel equilibrado para la mayoría de las cargas de trabajo empresariales.
  • Usa una arquitectura con cómputo por un lado y almacenamiento remoto por otro.
  • Eso significa que el proceso SQL corre en un nodo de cómputo, pero los archivos de datos y log residen en una capa de almacenamiento separada.
  • Se apoya en almacenamiento remoto premium y busca equilibrio entre costo, capacidad y operación.
  • Es la opción natural cuando no necesitas la latencia más baja ni una arquitectura extrema de escalado.

Business Critical

  • Está orientado a cargas OLTP (Online Transaction Processing, procesamiento transaccional en línea) con transacciones intensivas y requisitos de baja latencia.
  • Aquí el proceso SQL y el almacenamiento activo están en el mismo nodo, usando SSD local.
  • Eso reduce el tiempo que tarda el motor en leer y escribir, porque no depende tanto de una capa remota para el acceso principal al dato.
  • Usa varias réplicas de alta disponibilidad y una réplica legible para descarga de consultas de solo lectura.
  • Su principal valor no es solo soportar más datos, sino responder más rápido y recuperarse mejor ante fallos.

Hyperscale

  • Está diseñado para cargas que requieren crecimiento muy alto de almacenamiento y escalado flexible.
  • Su arquitectura desacopla cómputo y almacenamiento para escalar de forma independiente.
  • Permite volúmenes muy grandes de datos y escalado de lectura con varias réplicas según la necesidad.
  • No compite con Business Critical por el mismo enfoque: Hyperscale prioriza elasticidad y tamaño; Business Critical prioriza latencia e I/O (entrada/salida de datos a almacenamiento).
NivelEnfoque principalArquitectura / almacenamientoPunto fino
General Purpose Balance entre costo y capacidad para la mayoría de workloads El motor corre en nodos de cómputo y los archivos viven en almacenamiento remoto premium Es el perfil estándar; no busca la latencia más baja del portafolio.
Business Critical Baja latencia, alta I/O y recuperación rápida Proceso y datos activos juntos en nodos con SSD local, dentro de un clúster con varias réplicas Microsoft lo orienta a escenarios con respuesta de almacenamiento de alrededor de 1-2 ms y lectura descargada a réplica secundaria.
Hyperscale Crecimiento extremo, escalado flexible y lectura distribuida Almacenamiento desacoplado con caché SSD local por réplica de proceso Puede crecer hasta volúmenes muy altos y no está limitado por el patrón clásico de un solo nodo de datos.
Lectura rápida: si la prioridad es costo equilibrado, normalmente miras General Purpose. Si la prioridad es latencia y rendimiento transaccional, miras Business Critical. Si la prioridad es escalar almacenamiento y cómputo a gran volumen, miras Hyperscale.
Forma simple de recordarlo: en General Purpose el motor y los archivos están "separados" y se comunican entre capas; en Business Critical el motor y el almacenamiento principal están "pegados" en el mismo nodo de trabajo. Esa cercanía es la que ayuda a bajar la latencia y mejorar la respuesta transaccional.
Punto fino del portal: cuando en Azure SQL Database ves estas opciones de General Purpose, Business Critical e Hyperscale, normalmente estás en el modelo vCore. En el modelo DTU, la lógica de selección usa otras familias de servicio y no esta misma clasificación.

Seguridad, identidad y protección del dato

Red y exposición

  • Firewall: permite restringir acceso solo a IPs autorizadas.
  • Private Link / Private Endpoint: elimina la necesidad de exponer la base con IP pública.
  • La recomendación empresarial es acercar el acceso a la VNet del backend y no abrir Internet por defecto.

Autenticación

  • Conviene priorizar Microsoft Entra ID sobre usuarios SQL tradicionales.
  • El servidor lógico centraliza la integración con identidades corporativas.
  • Esto simplifica gobierno, auditoría y revocación de acceso.

Managed Identity

  • App Service, Functions, Container Apps o VM pueden autenticarse sin guardar contraseñas.
  • La identidad administrada se registra en Entra ID y se autoriza en la base.
  • Evita secretos embebidos en código, archivos o variables de entorno.

Protección del dato: en reposo, en tránsito y en uso

Proteger una base de datos no solo significa controlar quién entra. También implica asegurar que el dato permanezca protegido si alguien roba un disco, si intercepta una comunicación o incluso si un administrador tiene permisos amplios pero no debería ver datos sensibles en claro.

Además, el diseño de protección cambia según la industria. En sectores como banca, salud o gobierno, no basta con cifrar la base: también hay que definir qué perfil puede ver qué dato, con qué nivel de detalle y bajo qué contexto operativo.

Encriptación en reposo

  • Transparent Data Encryption (TDE): cifra los archivos de datos, logs y respaldos en reposo.
  • La idea práctica es que si alguien accediera al medio físico o al almacenamiento subyacente, no podría leer la información en claro.
  • Es la respuesta clásica al escenario extremo de "robarse el disco".

Encriptación en tránsito

  • La comunicación entre cliente y base debe viajar cifrada, normalmente mediante TLS.
  • Esto reduce el riesgo de lectura o manipulación del tráfico en la red.
  • En arquitectura enterprise no basta con cerrar firewall; también debe cuidarse el canal.

Dato sensible frente a humanos

  • Always Encrypted: protege columnas sensibles para que ciertos datos ni siquiera sean visibles en claro para el motor o para administradores no autorizados.
  • Dynamic Data Masking: oculta parcialmente el dato en consultas para ciertos perfiles, pero no sustituye la encriptación.
  • La diferencia clave es que enmascarar no equivale a cifrar.

Gobierno del dato y segregación de funciones

Diseño por perfil

  • No solo se diseñan tablas, relaciones e índices; también se diseña quién puede consultar cada atributo.
  • Un mismo registro puede contener datos operativos, sensibles y regulatorios con reglas de acceso distintas.
  • La pregunta correcta no es solo "quién entra a la base", sino "qué columnas y qué filas debería ver cada rol".

Ejemplo en salud

  • Un médico sí puede necesitar ver datos clínicos en claro para atender a un paciente.
  • Eso no significa que todos los médicos deban ver todos los pacientes sin restricción.
  • Ahí entran controles como diseño por perfiles, filtros por contexto, Row-Level Security y reglas de negocio a nivel aplicación y datos.

Rol del administrador

  • El administrador de base de datos debería enfocarse en estructura, rendimiento, respaldos, usuarios y operación.
  • En un modelo maduro, no debería ser consumidor natural del contenido sensible almacenado en las tablas.
  • Por eso la protección del dato no es solo técnica; también es un tema de gobierno y separación de responsabilidades.

Cuando la pregunta es "ni el administrador de base debe ver ciertos valores sensibles", normalmente la respuesta va hacia protección de columnas, no solo a firewall o TLS.
Lectura de arquitectura: en industrias reguladas, la seguridad del dato no se resuelve con un solo mecanismo. Normalmente combina cifrado, enmascaramiento, restricción por filas o perfiles y una separación clara entre quien administra la plataforma y quien está autorizado a consumir la información.

SQL, Data Lake y analítica: no cumplen el mismo papel

Otra confusión frecuente es pensar que una base SQL y un Data Lake compiten por el mismo problema. En realidad responden a necesidades distintas.

Azure SQL Database

  • Está orientado a datos estructurados y operaciones transaccionales.
  • Es ideal para aplicaciones que insertan, actualizan, validan y consultan datos del negocio en tiempo casi real.
  • Su foco principal es integridad, consistencia y soporte a la operación de la aplicación.

Data Lake

  • Se usa para almacenar grandes volúmenes de datos crudos, históricos o semiestructurados.
  • Es común en escenarios de analítica, ciencia de datos, machine learning o integración masiva.
  • No reemplaza una base transaccional; la complementa.

Analítica y Big Data

  • Cuando el objetivo es explotar datos a gran escala, cruzar fuentes o procesar históricos, suelen intervenir servicios analíticos encima del lake.
  • La aplicación suele escribir en SQL; la analítica suele consolidar y procesar fuera del patrón OLTP.
  • Es normal que SQL alimente procesos hacia un Lake para reporting, BI o modelos avanzados.

Alta disponibilidad y réplicas

Alta disponibilidad base

  • En PaaS, Microsoft ofrece disponibilidad integrada como parte del servicio administrado.
  • Esto reduce la complejidad frente a montar clústeres manuales en VM.

Geo-replicación

  • Permite mantener una réplica en otra región.
  • Sirve para disaster recovery y también para descargar consultas de lectura.
  • Un patrón común es enviar reportes pesados a la réplica y dejar la primaria para transacciones.

Failover Groups

  • Automatizan la conmutación entre una base principal y su secundaria en otra región.
  • La ventaja clave es que la aplicación se conecta a un endpoint estable, no directamente al nombre fijo de una sola base o servidor.
  • Si la región primaria falla, Azure redirige el punto de conexión hacia la secundaria y evita que el equipo tenga que cambiar manualmente la cadena de conexión en plena contingencia.
Forma simple de entenderlo: una geo-réplica te da una copia en otra región. Un Failover Group agrega la lógica de conmutación y el nombre de conexión estable para que la aplicación no tenga que saber en qué región está activa la base en ese momento.
ConceptoQué haceValor práctico
Geo-replicación Mantiene una copia sincronizada de la base en otra región. Sirve para disaster recovery y, en algunos diseños, para descargar lectura.
Failover Group Orquesta el cambio entre primaria y secundaria y expone un endpoint estable. Evita editar manualmente cadenas de conexión cuando ocurre una caída regional.

Ejemplo práctico: e-commerce

  • La base principal está en East US y la secundaria en Central US.
  • La aplicación web no apunta a `servidor-east.database.windows.net`, sino al endpoint del Failover Group.
  • Si East US tiene una caída, Azure promueve la secundaria y la aplicación sigue usando el mismo nombre de conexión.
  • El resultado es menor tiempo de interrupción y menos trabajo manual del equipo en una crisis.

Ejemplo práctico: banca o salud

  • Un sistema transaccional no puede depender de que alguien edite configuraciones a mano durante un incidente.
  • Con Failover Group, el plan de continuidad ya deja preparada la ruta de cambio hacia la región secundaria.
  • Eso no elimina la necesidad de validar consistencia, pruebas de contingencia y tiempos de recuperación, pero sí reduce mucho la fricción operativa.

Escalabilidad avanzada y arquitecturas distribuidas

Lectura vs. escritura

  • No siempre se balancean igual.
  • En muchas arquitecturas la primaria concentra escrituras y consistencia, mientras que las secundarias ayudan a absorber lecturas.
  • Eso mejora escalabilidad, pero también obliga a entender latencia y consistencia entre nodos.

Fragmentación o sharding

  • Cuando una sola base deja de ser suficiente, una estrategia es dividir los datos en varias bases más pequeñas.
  • La división puede hacerse por cliente, región, rango o criterio de negocio.
  • Esto incrementa la complejidad porque el dato deja de estar centralizado en un solo motor.

Sincronización y supervisión

  • En esquemas distribuidos suelen aparecer servicios intermedios de sincronización, integración o supervisión.
  • Ya no basta con diseñar tablas; también hay que diseñar flujos de consistencia, reconciliación y monitoreo.
  • Mientras más extrema es la distribución, más importante se vuelve la observabilidad del dato.
Lectura arquitectónica: las réplicas ayudan mucho para lectura, continuidad y disaster recovery. Cuando el reto real es la escritura distribuida o el volumen extremo, la conversación deja de ser solo "poner más réplicas" y pasa a ser de particionado, sincronización y gobierno del dato.

SQL Data Sync for Azure

SQL Data Sync es un servicio de Azure SQL Database pensado para sincronizar los datos seleccionados entre varias bases, incluso combinando Azure SQL Database y SQL Server on-premises. Su modelo no es de clúster ni de alta disponibilidad; es un modelo de sincronización de datos.

Cómo funciona

  • Trabaja con un esquema de hub and spoke.
  • La base hub debe ser Azure SQL Database.
  • Los miembros pueden ser otras Azure SQL Database o instancias de SQL Server.
  • La sincronización ocurre entre el hub y cada miembro, no entre todos los nodos de forma directa.

Casos donde encaja

  • Sincronizar subconjuntos de tablas entre una base en Azure y sucursales o sedes que aún tienen SQL Server local.
  • Mantener datos operativos compartidos entre varias bases cuando no se busca un clúster transaccional estricto.
  • Escenarios legacy donde el objetivo es propagar datos, no montar una estrategia completa de HA/DR.

Límites importantes

  • No ofrece consistencia transaccional estricta; su modelo es de consistencia eventual.
  • Usa triggers y tablas auxiliares para detectar cambios, lo que puede impactar rendimiento.
  • No es una herramienta de backup, restore, ni una solución de disaster recovery.
  • No soporta Azure SQL Managed Instance.
¿Qué significa hub and spoke? Es un patrón donde existe un nodo central llamado hub y varios nodos periféricos llamados spokes. En SQL Data Sync, la base hub es el punto central de sincronización y cada base miembro se conecta con ella. La comunicación ocurre entre hub ↔ member, no entre todos los miembros directamente.

Ejemplo simple

  • Una base central en Azure se define como hub.
  • Tres sucursales con sus propias bases se definen como spokes.
  • La sucursal A sincroniza con el hub.
  • La sucursal B sincroniza con el hub.
  • La sucursal C sincroniza con el hub.
  • La sucursal A no sincroniza directamente con B o C.

Por qué se usa

  • Centraliza el control y la coordinación del intercambio de datos.
  • Facilita reglas de sincronización y resolución de conflictos.
  • Evita una malla completa de conexiones entre todas las bases.
  • Hace más simple la administración cuando hay varias sedes o sistemas satélite.

Transición habitual: cuando un equipo empieza a buscar alternativas a SQL Data Sync, es muy común que termine evaluando Azure Data Factory. La razón es que ADF permite mover y refrescar datos entre sistemas, pero con una lógica distinta: pipelines programados y procesos de integración, no sincronización bidireccional administrada entre bases como servicio dedicado.
Forma simple de recordarlo: si el problema es "quiero que ciertos datos se propaguen entre varias bases", SQL Data Sync entra en la conversación. Si el problema es "quiero continuidad operativa, failover automático o consistencia transaccional fuerte", entonces hay que mirar otras estrategias como réplicas, failover groups o mecanismos distintos de replicación.
Síntesis: en Azure SQL la decisión correcta combina responsabilidad operativa, patrón de carga, identidad, protección del dato y estrategia de continuidad. Elegir entre VM y PaaS no es solo una cuestión de tecnología, sino de qué parte de la administración quiere seguir cargando el equipo y qué tan lejos necesita llevar la seguridad y la escalabilidad de los datos.

7.3 Azure Data Factory

Azure Data Factory (ADF) es un servicio PaaS de integración y orquestación de datos. No es una base de datos, no es un Data Lake y no es una herramienta de alta disponibilidad transaccional. Su función principal es mover, transformar y coordinar datos entre distintos orígenes y destinos, tanto en Azure como on-premises.

Por eso, ADF suele aparecer cuando alguien explora alternativas a Azure SQL Data Sync. No porque lo reemplace de forma idéntica, sino porque permite construir flujos de copia e integración que cubren parte de esa necesidad desde un enfoque más amplio de ETL / ELT (extraer, transformar y cargar / extraer, cargar y transformar) y orquestación.

Qué sí es

  • Un orquestador de pipelines de datos.
  • Una capa para copiar, transformar y programar flujos de integración.
  • Un puente entre SQL, Storage, Data Lake, Synapse, Databricks, APIs y orígenes on-premises.

Qué no es

  • No almacena los datos del negocio como lo haría una base.
  • No sustituye Azure SQL, Cosmos DB o Data Lake.
  • No resuelve por sí solo consistencia transaccional ni continuidad operativa de una base de datos.

Idea clave

  • Si Azure SQL es donde viven los datos transaccionales, ADF es una opción para moverlos o procesarlos hacia otro destino.
  • Su enfoque es ETL / ELT, integración y automatización de cargas.

Componentes principales

Pipeline

  • Es el flujo lógico completo.
  • Agrupa actividades, dependencias, parámetros y triggers.

Activity

  • Es la acción concreta: copiar, ejecutar un notebook, correr un stored procedure o validar datos.
  • La más conocida es Copy Activity.

Linked Service / Dataset

  • Linked Service: define la conexión al origen o destino.
  • Dataset: representa el conjunto de datos específico que usará la actividad.

Integration Runtime

  • Es la capa de ejecución que mueve o despacha el trabajo.
  • Puede ser Azure IR, Self-hosted IR o Azure-SSIS IR.

Cuándo encaja Azure Data Factory

Ingesta y copias

  • Copiar datos desde SQL Server on-premises hacia Azure SQL, Blob Storage o Data Lake.
  • Centralizar datos dispersos en varias fuentes Azure u on-premises.

Orquestación

  • Programar pipelines por horario, eventos o dependencias.
  • Encadenar pasos de extracción, transformación, carga y notificación.

Analítica y modernización

  • Llevar datos de sistemas transaccionales hacia Data Lake o plataformas analíticas.
  • Alimentar procesos en Synapse, Databricks o modelos de BI.
EscenarioRol de ADFEjemplo práctico
Migración de datos Mueve información entre orígenes y destinos Copiar tablas desde SQL Server on-premises hacia Azure SQL Database.
Data Lake / BI Ingiere y publica datos hacia almacenamiento analítico Extraer datos diarios de ERP y escribirlos en ADLS Gen2 en formato parquet.
Orquestación empresarial Coordina procesos entre múltiples servicios Correr una copia, luego un notebook en Databricks y al final un procedimiento de publicación.

Integration Runtime: el punto operativo más importante

Azure IR

  • Se usa cuando el movimiento y la ejecución pueden resolverse desde Azure.
  • Es el caso más común para fuentes y destinos cloud.

Self-hosted IR

  • Se instala en una máquina propia o VM para llegar a redes privadas u on-premises.
  • Es clave cuando el origen o destino no es públicamente accesible.

Azure-SSIS IR

  • Sirve para ejecutar paquetes SSIS (SQL Server Integration Services) en modo administrado.
  • Es útil en migraciones donde ya existe una inversión fuerte en Integration Services.

Relación con Azure SQL, Data Lake y sincronización

ADF vs Azure SQL

  • Azure SQL almacena y sirve datos transaccionales.
  • ADF mueve, transforma u orquesta lo que ocurre alrededor de esos datos.

ADF vs Data Lake

  • Data Lake es el repositorio de datos masivos.
  • ADF puede ser el mecanismo que carga o refresca ese lake.

ADF vs SQL Data Sync

  • ADF está orientado a pipelines y procesos de integración.
  • SQL Data Sync está orientado a sincronización de datos entre bases, con consistencia eventual.
  • ADF suele aparecer como alternativa moderna cuando el objetivo real es refrescar, copiar o consolidar datos entre sistemas.
  • No resuelven exactamente el mismo problema.
Punto fino: usar ADF para copiar datos no significa que tengas sincronización en tiempo real ni replicación transaccional. Normalmente se trabaja con ejecuciones programadas, lotes o flujos orquestados, no con una base secundaria activa como en escenarios de alta disponibilidad.
Síntesis: Azure Data Factory es la pieza que conecta el mundo transaccional con el mundo analítico y de integración. No compite con una base de datos; la complementa al encargarse del movimiento, transformación y orquestación de datos entre sistemas.

8. Seguridad e identidad

La seguridad en Azure no se limita a usuarios humanos. Una parte central del diseño es cómo se autentican los servicios entre sí sin dejar secretos embebidos en código, archivos de configuración o variables de entorno. Ahí entra Managed Identity.

Fundamento

  • Autenticación: verifica quién eres; autorización define qué puedes hacer.
  • Entra ID: centraliza identidades, grupos, aplicaciones y tokens.
  • RBAC: asigna permisos sobre recursos Azure según alcance, rol y necesidad operativa.
  • Managed Identity: es una identidad de sistema para un recurso Azure, pensada para autenticarse sin credenciales embebidas.

System-assigned

  • La identidad se crea directamente sobre el recurso Azure.
  • La relación práctica es 1:1: una identidad para un recurso.
  • Su ciclo de vida está vinculado al recurso; si el recurso se elimina, la identidad también.
  • Conviene en soluciones simples o cuando no se requiere compartir la misma identidad entre varios servicios.

User-assigned

  • La identidad se crea como un recurso independiente y luego se vincula a uno o varios servicios.
  • La relación práctica es 1:N: una identidad puede ser usada por múltiples recursos.
  • No depende del ciclo de vida de un solo servicio; por eso ofrece más control administrativo.
  • Es la opción adecuada cuando varias interacciones entre servicios deben compartir permisos específicos y gobernanza más fina.
TipoRelaciónCuándo convienePunto fino
System-assigned managed identity 1:1 Recursos aislados, soluciones sencillas o dependencias puntuales. Si se elimina el recurso, también desaparece la identidad.
User-assigned managed identity 1:N Soluciones con varios servicios, permisos compartidos o necesidad de administrar una identidad de forma separada. Debe crearse previamente y luego asociarse explícitamente a los recursos que la usarán.

Uso en código y práctica operativa

Bibliotecas y credenciales

  • Las SDK (bibliotecas oficiales de desarrollo) modernas suelen ofrecer un mecanismo por defecto como DefaultAzureCredential.
  • En desarrollo es útil porque intenta varias rutas de autenticación: variables de entorno, sesión del usuario, herramientas locales y managed identity.
  • En producción no siempre es la mejor práctica dejar la selección al mecanismo por defecto si existen varias opciones habilitadas.

Recomendación

  • En ambientes productivos conviene usar una identidad administrada explícita, no una resolución ambigua.
  • Si el servicio tiene varias posibilidades de autenticación, se debe indicar claramente cuál identidad debe utilizarse.
  • Esto evita conflictos entre identidad del sistema, identidad asignada por usuario, variables de entorno u otros mecanismos del host.

Migraciones y compatibilidad

  • No basta con que el servicio Azure soporte managed identity; también la librería usada por la aplicación debe ser compatible.
  • En migraciones de versiones puede ser necesario actualizar dependencias, cambiar configuración o incluso modificar código.
  • Siempre debe revisarse compatibilidad de SDK, drivers y versiones del runtime.

Casos prácticos de integración

Key Vault

  • Para leer secretos, certificados o llaves, el recurso consumidor necesita una identidad administrada con permisos sobre la bóveda.
  • No basta con que el secreto exista; la identidad debe estar vinculada y autorizada.

Container Apps y ACR

  • Cuando una Container App necesita extraer imágenes desde Azure Container Registry, puede requerir una identidad para autenticarse contra el registry.
  • Esto evita guardar credenciales del registro dentro de la configuración del despliegue.

Azure SQL

  • Una cosa es habilitar la identidad en el servicio consumidor; otra distinta es darla de alta como usuario dentro del motor de la base de datos.
  • La conexión con Entra ID requiere configuración tanto en el servicio como en la capa de SQL.
Síntesis: Managed Identity elimina secretos embebidos, pero no elimina el diseño. Hay que decidir si la identidad vive y muere con el recurso (system-assigned) o si debe existir como activo compartido y administrable (user-assigned), además de validar permisos, código y compatibilidad de librerías.
Advertencia práctica: que Azure soporte identidades administradas no significa que cualquier versión de la librería del proyecto las use correctamente. En cambios de arquitectura o migraciones, revisar dependencias es parte del trabajo, no un detalle menor.

9. Key Vault

Azure Key Vault es el servicio diseñado para almacenar de forma segura secretos, certificados y claves criptográficas. En la práctica se usa para guardar elementos sensibles como contraseñas, API keys, connection strings, certificados TLS o llaves de cifrado, de manera que no queden embebidos en código, archivos de configuración o variables expuestas.

Qué guarda

  • Secrets: contraseñas, tokens, API keys, cadenas de conexión.
  • Certificates: certificados y su ciclo de vida.
  • Keys: llaves criptográficas para escenarios de cifrado, firma o integración con otros servicios.

Qué resuelve

  • Evita que las credenciales vivan dentro del código fuente.
  • Centraliza control, auditoría y caducidad de valores sensibles.
  • Permite que los servicios consuman secretos mediante identidad, no mediante usuarios embebidos.

Idea clave

  • Guardar un secreto en Key Vault no resuelve por sí solo la seguridad.
  • Lo importante es quién puede administrarlo y quién puede leer su valor.

Separación de permisos: control plane vs data plane

En Key Vault hay una separación muy importante entre administrar el recurso y consumir el contenido. Esa diferencia suele ser donde aparecen los errores de diseño.

CapaQué permiteEjemplo de rol o acciónPunto fino
Control plane Crear el vault, cambiar configuración, redes, tags y propiedades del recurso Administrar el recurso Azure Key Vault Administrar el recurso no implica poder leer el valor de los secretos.
Data plane Leer, crear, actualizar o borrar secretos, claves y certificados Roles como Key Vault Secrets User o Key Vault Secrets Officer Aquí se define quién realmente puede consumir o manipular el contenido sensible.

Administrador del recurso

  • Puede configurar la bóveda, sus redes, diagnósticos o su modelo de acceso.
  • No debería asumir por defecto que puede leer todos los secretos.
  • En un diseño sano, administrar la plataforma y consumir secretos son funciones distintas.

Administrador de secretos

  • Puede crear, modificar o eliminar secretos.
  • Es distinto de quien administra el recurso completo.
  • En entornos gobernados conviene separar esta función del dueño de la aplicación y del operador de infraestructura.

Consumidor del secreto

  • Es el servicio o aplicación que necesita leer el valor para operar.
  • Lo recomendado en Azure es que ese consumo se haga con Managed Identity y permisos de lectura mínimos.
  • La identidad no necesita administrar secretos; solo leer los que realmente usa.

Managed Identity y consumo seguro

Patrón recomendado

  • Un servicio como App Service, Functions o Container Apps obtiene una Managed Identity.
  • Esa identidad recibe permisos de consumo sobre el vault.
  • La aplicación lee el secreto sin guardar credenciales en su código.

Permisos mínimos

  • Si la aplicación solo debe leer secretos, no necesita permisos de escritura o borrado.
  • Un rol como Key Vault Secrets User suele ser suficiente para consumo.
  • Esto ayuda a respetar principio de menor privilegio.

Error común

  • Dar permisos amplios al recurso o al equipo sin distinguir consumidor, operador y administrador.
  • El resultado es que demasiadas identidades terminan con acceso a datos sensibles.

Caducidad, rotación y versionamiento

En una organización gobernada, los secretos no deberían vivir indefinidamente. Key Vault soporta expiración, versionado y, según el tipo de objeto y la integración, estrategias de rotación.

Expiración

  • Los secretos pueden tener fecha de vencimiento.
  • Esto ayuda a imponer disciplina de renovación y cumplimiento.
  • Es especialmente relevante para API keys, certificados y credenciales compartidas.

Versionamiento

  • Cuando actualizas un secreto, Key Vault genera una nueva versión.
  • Eso permite cambiar valores sin perder el historial operativo del objeto.
  • El reto arquitectónico es decidir si los consumidores apuntan a una versión fija o a la versión más reciente.

Rotación

  • La rotación busca reemplazar periódicamente secretos o llaves antes de que caduquen o queden expuestos.
  • En llaves criptográficas existe soporte de rotación automática con políticas específicas.
  • En secretos, la renovación muchas veces depende del proceso de la aplicación o de automatizaciones externas.

Ejemplo práctico con Container Apps

Referencia a Key Vault

  • Container Apps puede consumir secretos desde Key Vault mediante una identidad administrada.
  • La configuración usa un URI del secreto.
  • Ese URI puede apuntar a una versión específica o a la versión más reciente.

Impacto de la versión

  • Si usas un URI sin versión, el servicio puede recoger automáticamente la versión más reciente.
  • Si usas un URI con versión fija, controlas exactamente qué valor consume la aplicación.
  • La decisión depende de si quieres prioridad en estabilidad o en actualización automática.
Punto fino: una cosa es publicar una nueva versión del secreto en Key Vault y otra es que todos los consumidores la tomen inmediatamente. Siempre hay que revisar cómo refresca secretos cada servicio y si la referencia es versionada o no.
Síntesis: Key Vault no es solo un repositorio de secretos. Es una pieza de gobierno: separa administración del recurso, administración del contenido y consumo por servicios. Su valor real aparece cuando se combina con Managed Identity, permisos mínimos, expiración, rotación y una estrategia clara de versionamiento.

10. Networking y redes virtuales

La red en Azure no solo conecta recursos; define aislamiento, alcance, seguridad y la forma en que una solución entra, sale o se integra con otros sistemas. La pieza base es la Virtual Network (VNet), una red privada lógica dentro de una región de Azure sobre la que segmentamos subredes, aplicamos políticas de tráfico y conectamos servicios privados.

10.1 VNet y espacio de direcciones

  • VNet: representa una red privada en la nube para recursos Azure y redes conectadas.
  • Rango CIDR: debe planearse desde el inicio; CIDR es la forma estándar de expresar bloques de direcciones IP, por ejemplo `10.0.0.0/16`, y no debe traslaparse con otras VNets ni con on-premises si habrá peering, VPN o ExpressRoute.
  • Alcance regional: una VNet pertenece a una región, aunque puede conectar recursos zonales y no zonales dentro de ella.
  • Subredes: permiten separar capas como frontend, integración, datos y administración.
  • IP reservadas: Azure reserva las primeras 4 IPs y la última IP en cada subnet; no deben contarse como capacidad usable.

10.2 Control del tráfico

  • NSG: los Network Security Groups permiten reglas allow/deny de entrada y salida por origen, destino, puerto y protocolo.
  • Route tables / UDR: las User Defined Routes cambian el siguiente salto del tráfico para forzarlo hacia firewalls, NVAs (appliances virtuales de red), gateways o salidas específicas.
  • Rutas del sistema: Azure crea rutas por defecto entre subnets, hacia Internet y hacia prefijos privados conocidos.
  • Segmentación real: la subnet define frontera lógica; el NSG y la tabla de rutas definen qué puede cruzarla y por dónde.

10.3 Acceso privado a PaaS

  • Service endpoints: extienden la identidad de la subnet hacia servicios Azure y mantienen el tráfico en el backbone de Microsoft.
  • Private endpoints: crean una NIC con IP privada dentro de la VNet para acceder a servicios PaaS sin exponerlos a Internet.
  • Recomendación actual: Microsoft prioriza Private Link cuando se requiere aislamiento fuerte, acceso privado desde on-premises o reducir exfiltración.
  • DNS privado: al usar private endpoints normalmente se requiere Private DNS Zone, una zona DNS interna, para resolver el nombre del servicio a su IP privada.

10.4 Conectividad entre redes

  • VNet peering: conecta VNets por la red troncal de Microsoft con baja latencia y sin gateway intermedio.
  • No transitivo: si A está emparejada con B y B con C, A no alcanza a C automáticamente.
  • VPN Gateway: conecta Azure con on-premises o con otras VNets mediante túneles cifrados sobre Internet.
  • ExpressRoute: extiende la red corporativa a Azure por un enlace privado dedicado, con mayor predictibilidad que una VPN sobre Internet.

10.5 Publicación y salida a Internet

  • Public IP: expone un recurso o balanceador hacia clientes externos cuando la arquitectura lo requiere.
  • NAT Gateway: centraliza y estabiliza la salida a Internet de una subnet sin asignar IP pública a cada recurso.
  • Application Gateway / Load Balancer: publican aplicaciones o distribuyen tráfico, pero no sustituyen la segmentación de red.
  • Principio práctico: preferir salida controlada y acceso privado antes que exponer recursos directamente.

10.6 Diseño y buenas prácticas

  • Separar por función: front, backend, integración, datos y administración no deberían compartir la misma subnet por defecto.
  • Subnets delegadas: algunos servicios PaaS necesitan permisos explícitos sobre la subnet para desplegarse correctamente.
  • Hub-and-spoke: patrón común para centralizar DNS, firewalls, VPN/ER y servicios compartidos.
  • Plan de nombres y direcciones: conviene diseñarlo junto con la arquitectura, porque cambiar rangos o peerings después suele ser costoso.

11. Observabilidad

La observabilidad en Azure no es un solo servicio. Es un conjunto de capacidades para ver, diagnosticar, alertar y reaccionar ante el comportamiento de aplicaciones, infraestructura y servicios administrados. La pieza central es Azure Monitor, pero alrededor de él existen experiencias y servicios complementarios que cubren infraestructura, aplicaciones, salud de plataforma y recomendaciones operativas.

Azure Monitor

  • Es la plataforma unificada de monitoreo de Azure.
  • Centraliza métricas, logs, eventos y trazas desde recursos Azure, otros clouds y entornos on-premises.
  • Es el punto base para alertas, dashboards, consultas y automatización operativa.

Application Insights

  • Es la capacidad de APM (Application Performance Monitoring, monitoreo del rendimiento de aplicaciones) dentro de Azure Monitor para aplicaciones.
  • Sirve para requests, dependencias, excepciones, tiempos de respuesta, mapas de aplicación y trazabilidad.
  • Ayuda a ver qué pasa dentro del código y en la experiencia real de la app.

Log Analytics Workspace

  • Es el almacén donde Azure Monitor guarda y consulta logs.
  • Permite centralizar datos de múltiples recursos y hacer consultas con KQL (Kusto Query Language, lenguaje de consulta de Azure Monitor y Log Analytics).
  • Es uno de los componentes más comunes y estándares en diseños de monitoreo en Azure.

Alertas: no es un servicio aislado, es una capacidad transversal

Metric Alerts

  • Se disparan con base en métricas ya calculadas.
  • Son útiles para CPU, memoria, latencia, disponibilidad o conteos.

Log Alerts

  • Se basan en consultas sobre logs.
  • Permiten detectar patrones más complejos, correlaciones o errores específicos.

Health Alerts

  • Aquí entran Service Health alerts y Resource Health alerts.
  • No son otro producto distinto: son tipos de reglas de alerta apoyadas en señales de salud.
Aclaración importante: lo que mencionas como "alerta de salud" efectivamente no suele verse como un servicio aislado de monitoreo, sino como una regla de alerta construida sobre Azure Service Health o Azure Resource Health para notificar incidentes, mantenimientos o degradaciones.

Service Health vs Resource Health vs Advisor

Azure Service Health

  • Mira la salud de Azure desde la perspectiva de la plataforma y de tu suscripción.
  • Te avisa si un servicio o región que usas está teniendo incidentes o mantenimiento planeado.
  • Es muy útil para saber si el problema viene de Azure y no de tu configuración.

Azure Resource Health

  • Baja al nivel del recurso individual.
  • Sirve para ver si una VM, App Service o SQL Database estuvo disponible o fue afectado por un problema del proveedor.
  • Ayuda mucho en diagnóstico de incidentes y revisión de SLA.

Azure Advisor

  • No es monitoreo en tiempo real como tal.
  • Es un asistente de recomendaciones que analiza configuración y telemetría de uso.
  • Se usa para mejorar costo, seguridad, rendimiento, confiabilidad y excelencia operativa.

Otros componentes comunes de una estrategia de monitoreo

Action Groups

  • Definen qué hacer cuando una alerta se dispara.
  • Pueden enviar correo, SMS, push, webhooks o disparar automatizaciones.

Workbooks

  • Permiten construir vistas personalizadas con consultas, gráficas y diagnósticos.
  • Son muy comunes para tableros operativos o troubleshooting guiado.

Diagnostic settings / DCR

  • Definen hacia dónde se envían logs y métricas.
  • Son el punto base para que los datos lleguen a Log Analytics, Storage o Event Hub. En escenarios modernos también aparecen las DCR (Data Collection Rules, reglas de recolección de datos).
Lectura arquitectónica: para un diseño básico y serio en Azure, lo mínimo común suele ser Azure Monitor + Log Analytics Workspace + Alerts + Service/Resource Health, y si hay aplicaciones propias, también Application Insights. Azure Advisor entra como complemento de recomendaciones, no como reemplazo del monitoreo operativo.

12. Costos

Hablar de costos en Azure no es solo revisar precios por servicio. También implica entender quién ve la facturación, quién administra el pago, cómo se monitorea el gasto y cómo se estima una implementación antes de desplegarla. En la práctica conviene separar dos planos: el operativo y el de facturación.

Factores de costo

  • Tiempo de ejecución, horas activas o instancias aprovisionadas.
  • Almacenamiento, transacciones, transferencia de datos y redundancia.
  • SKU, región, elasticidad, reservas y modelo de compra.

Costo operativo

  • Es el gasto real del uso diario de los recursos desplegados.
  • Se analiza con Cost Management, presupuestos, alertas y tendencias de consumo.
  • Su foco es controlar desviaciones y optimizar el gasto ya existente.

Costo estimado

  • Es el costo proyectado de una arquitectura que aún no está desplegada o que se está rediseñando.
  • Se trabaja con la Azure Pricing Calculator.
  • Su foco es anticipar presupuesto y validar viabilidad financiera.

Facturación: billing account, billing profile y pago

En el modelo moderno de Microsoft Customer Agreement (MCA), la jerarquía de facturación separa el contrato general del detalle operativo de las facturas.

ConceptoQué representaPor qué importa
Billing account La cuenta de facturación principal del acuerdo Es el nivel alto donde se organiza la relación comercial y el acceso global de facturación.
Billing profile El agrupador que genera una factura y administra métodos de pago Aquí se ven facturas, cargos, moneda, dirección de cobro y pagos asociados.
Invoice section Subdivisión interna dentro del billing profile Sirve para separar cargos por proyecto, departamento o ambiente dentro de una misma factura.

Billing profile

  • Es uno de los conceptos más importantes de facturación moderna en Azure.
  • En términos prácticos, es el nivel donde se gestionan facturas y métodos de pago.
  • Es el equivalente funcional de donde se consolida lo que se va a cobrar.

Quién puede ver y quién puede pagar

  • No todo usuario con acceso técnico a la suscripción debe tener acceso a facturación.
  • Hay roles específicos de billing para ver facturas, cargos, usage y gestionar pagos.
  • En una organización madura, finanzas, compras o IT managers suelen tener estos accesos, no cualquier operador de recursos.

Métodos de pago

  • Se administran a nivel de facturación, no como parte normal del despliegue de recursos.
  • El billing profile es donde se asocia la forma de pago para las facturas correspondientes.
  • Esto debe separarse claramente de los permisos para crear recursos o administrar infraestructura.

Control de costos: monitoreo, presupuestos y alertas

Cost Management

  • Es la herramienta principal para analizar costos reales, tendencias y distribución del gasto.
  • Permite filtrar por suscripción, grupo de recursos, tags, servicios y periodos.
  • Es la base de la gestión operativa del costo.

Presupuestos

  • Permiten definir un límite esperado de gasto por alcance y periodo.
  • No bloquean el consumo por sí solos, pero ayudan a gobernarlo.
  • Se usan como referencia para detectar desviaciones tempranas.

Alertas de presupuesto

  • Se disparan cuando el gasto real o proyectado alcanza umbrales como 50%, 80% o 100%.
  • Sirven para notificar y actuar antes de perder control del costo.
  • Pueden integrarse con grupos de acción y automatizaciones según el alcance.
HerramientaPara qué sirvePunto fino
Cost Analysis Analizar consumo real, tendencias y distribución de cargos Sirve para operación diaria y finops; no estima una arquitectura futura.
Budget Definir un límite esperado de gasto El presupuesto alerta; no necesariamente detiene el servicio.
Budget alerts Notificar cuando el gasto alcanza umbrales definidos Son básicos para gobierno, pero dependen de que alguien reaccione o de automatización complementaria.
Anomaly alerts Detectar cambios inesperados de consumo Ayudan a identificar picos o caídas anormales que no estaban presupuestados.
Azure Pricing Calculator Estimar el costo de una posible implementación Trabaja con supuestos y cantidades esperadas; no sustituye la factura real.
Lectura práctica: Cost Management responde "cuánto estoy gastando y en qué". La Pricing Calculator responde "cuánto podría costar si despliego esta arquitectura".

Asignación de costo por proyecto: tags vs suscripción vs factura

Esta es una confusión muy común en cursos y exámenes. Cuando la pregunta dice algo como "quiero adjudicar o imputar el costo por proyecto", la respuesta más habitual es tags, porque las etiquetas permiten clasificar, filtrar, agrupar y reportar el consumo por proyecto, centro de costo, ambiente o dueño.

Sin embargo, eso no significa que Azure vaya a emitir automáticamente una factura separada para cada proyecto solo por tener una etiqueta. Lo que hace Azure es ayudarte a identificar el gasto para luego hacer showback o chargeback interno dentro de la organización.

MecanismoPara qué sirveCuándo usarloLo que no resuelve por sí solo
Tags Asignar y analizar costos por proyecto, área, ambiente o centro de costo Cuando varios proyectos viven en la misma suscripción y necesitas repartir el costo de forma rápida y flexible No crea facturas separadas por proyecto; solo facilita el filtrado y reporte del gasto
Suscripción Aislar administración, permisos, cuotas, políticas y también costo Cuando quieres separación fuerte entre proyectos o ambientes, no solo visibilidad del costo Es una frontera administrativa más pesada; no siempre conviene crear una suscripción por cada proyecto
Billing profile / invoice section Organizar cargos dentro del ámbito de facturación en MCA Cuando también quieres que la estructura de la factura tenga separación por proyecto, departamento o unidad No sustituye tags para análisis granular de recursos individuales

Ejemplo práctico con tags

  • Asignas Proyecto = A a un conjunto de recursos y Proyecto = B a otro.
  • En Cost Management agrupas el gasto por esa etiqueta.
  • El sistema puede mostrar algo como: Proyecto A = 5 pesos y Proyecto B = 6 pesos.
  • Con eso ya puedes imputar internamente el costo al dueño de cada proyecto.

Lectura correcta del examen

  • Si la pregunta es "cómo asigno o identifico costo por proyecto", normalmente la respuesta es tags.
  • Si la pregunta fuera "cómo separo administrativamente o financieramente proyectos completos", la conversación puede llevar a suscripciones o a invoice sections.
  • La clave es distinguir entre analizar costo y separar estructura de facturación.
Idea clave: las tags son la mejor respuesta para gestión y asignación de costos por proyecto dentro de una misma estructura Azure. Pero el cobro interno al dueño del proyecto sigue siendo un proceso organizacional tuyo, apoyado por la información que Azure te permite filtrar y exportar.

Estimación previa con la Azure Pricing Calculator

Cuándo usarla

  • Antes de desplegar una solución nueva.
  • Cuando se compara una arquitectura contra otra.
  • Cuando se requiere justificar presupuesto antes de implementación.

Qué se modela

  • Servicio, SKU, región, cantidad de instancias, horas, almacenamiento, transferencia y reservas.
  • En cuentas autenticadas, puede reflejar precios negociados o descuentos del acuerdo.
  • Es una herramienta de planeación, no una garantía exacta del cobro final.

Qué pasa si no se paga

La falta de pago no impacta de inmediato del mismo modo en todas las cuentas, pero sí existe un ciclo de estados que es importante entender desde operación y gobierno.

Past Due

  • La suscripción sigue activa, pero existe un saldo pendiente.
  • Azure empieza a mostrar avisos para regularizar el pago.
  • Es la etapa donde aún se puede actuar sin que los recursos entren en suspensión.

Disabled / Warned

  • Si el problema no se atiende, la suscripción puede quedar advertida o deshabilitada.
  • En ese estado ya no se pueden crear o modificar recursos y varios servicios dejan de operar con normalidad.
  • Por ejemplo, VMs pueden desasignarse, IPs temporales liberarse y algunos servicios quedar suspendidos o solo lectura.

Deleted

  • Si la situación persiste, la suscripción puede terminar eliminada junto con sus recursos subyacentes.
  • La ventana exacta depende del tipo de suscripción y del motivo de deshabilitación.
  • Microsoft documenta que algunas suscripciones pueden permanecer deshabilitadas entre 1 y 90 días antes de eliminarse permanentemente.
Punto fino: técnicamente una persona puede administrar recursos Azure y no tener ningún acceso a facturas o métodos de pago. Esa separación es deseable. Pero también significa que, si no existe coordinación entre operación y finanzas, una suscripción puede entrar en estados de advertencia o deshabilitación sin que el equipo técnico lo atienda a tiempo.
Síntesis: en Azure el costo no solo se gobierna eligiendo bien servicios y SKUs. También se gobierna definiendo qué alcance factura, quién puede ver y pagar, cómo se monitorea el gasto real y cómo se estima una implementación antes de desplegarla.
