Vista rapida

1. Resumen ejecutivo

Dos planos que no deben mezclarse

  • Plano de identidad: Microsoft Entra ID administra usuarios, grupos, aplicaciones y politicas de autenticacion.
  • Plano de recursos: Azure Resource Manager aplica autorizacion sobre recursos mediante Azure RBAC.

Decisiones de arquitectura recomendadas

  • Jerarquia clara de management groups y subscriptions.
  • Asignar acceso a traves de grupos, no directamente a usuarios.
  • Privilegios JIT con PIM y controles adaptativos con Conditional Access.
  • Priorizar managed identities frente a secretos manuales.
La combinacion mas audit-able suele ser: Azure Policy para gobernanza, Azure RBAC para autorizacion, y PIM para limitar privilegios permanentes.
Fundamentos

2. Conceptos base

La precision conceptual evita errores operativos y de auditoria. Una suscripcion de Azure confia en un solo tenant de Entra ID, mientras que un tenant puede ser usado por multiples suscripciones.

Elemento Que es Uso principal Alcance
Tenant Directorio de identidades, grupos, apps y politicas. Autenticacion, identidad y gobierno del directorio. Base de confianza para las suscripciones.
Management group Contenedor sobre suscripciones. Aplicar RBAC y Azure Policy a escala. Jerarquia superior de gobernanza.
Subscription Unidad administrativa, de costos y cuotas. Separar ownership, ambientes y presupuesto. Depende de un management group o del root.
Resource group Contenedor logico de recursos. Organizacion y lifecycle. Dentro de una suscripcion.
Role assignment Principal + rol + scope. Autorizar acciones sobre recursos. Management group, subscription, RG o recurso.
Security Principals

3. Tipos de identidades

Usuario interno

Miembro del tenant. Representa empleados o colaboradores internos.

Invitado B2B

Identidad externa que colabora en tu tenant, normalmente autenticada por su organizacion origen.

Service principal

Identidad operativa de una aplicacion dentro del tenant. Es quien realmente accede a recursos.

Managed identity

Identidad administrada por Azure para workloads. Elimina manejo manual de secretos o certificados.

No todas las identidades deben tratarse igual. Las identidades no humanas requieren estandares de credenciales, monitoreo y minimo privilegio igual o mas estricto que los usuarios.
Flujo

4. Flujo de autenticacion y autorizacion

En el modelo moderno de Azure, autenticacion y autorizacion se apoyan principalmente en OAuth 2.0 y OpenID Connect. Conditional Access actua como motor Zero Trust evaluando senales antes de emitir tokens.

1. Usuario Inicia acceso desde web, movil, portal o CLI.
2. Aplicacion Redirige a Entra ID para autenticacion.
3. Entra ID Evalua identidad y contexto de la solicitud.
4. Conditional Access Permite, bloquea o exige MFA y controles extra.
5. Token Se emite ID token y/o access token.
6. Recurso La API valida el token y aplica autorizacion.
Los logs de sign-in, audit y Activity Log permiten reconstruir este flujo durante auditorias o investigacion forense.
Control de acceso

5. Azure RBAC vs roles de Entra

Dimension Azure RBAC Roles de Entra ID
Que controla Recursos de Azure: VMs, storage, Key Vault, redes, etc. Objetos y configuracion del directorio: usuarios, grupos, apps y politicas.
Modelo Asignacion por scope. Roles administrativos del directorio.
Scopes tipicos Management group, subscription, resource group, recurso. Tenant, administrative unit o app registration.
Error comun Asignar Contributor u Owner demasiado arriba en la jerarquia. Creer que un rol de directorio da acceso a recursos Azure.
Un Global Administrator no reemplaza una asignacion RBAC, y un Owner de Azure no administra por si solo el directorio. Son planos distintos.
Modelo operativo

6. Grupos, roles y mejores practicas

Usar grupos como unidad de asignacion

  • Reduce errores y facilita auditoria.
  • Evita permisos directos a usuarios individuales.
  • Permite mover personas sin tocar cada recurso.

Role-assignable groups

  • Utiles para roles de Entra ID.
  • La propiedad se define al crear el grupo.
  • No es mutable despues.

Roles custom

  • Usar IDs de rol en automatizacion.
  • Evitar comodines porque pueden otorgar permisos futuros no previstos.
  • Disenar siempre con least privilege.
Tipo de grupo Uso principal Nota practica
Security group Autorizacion y control de acceso. Es la base recomendada para RBAC.
Microsoft 365 group Colaboracion en Teams, SharePoint y correo. No debe confundirse con un patron puro de autorizacion tecnica.
Distribution group Mensajeria. Mas orientado a Exchange que a autorizacion.

Patron recomendado de asignacion por alcance

Esta es una de las ideas mas importantes de la presentacion: en Azure no se da acceso general a una persona, se le da un rol especifico sobre un scope especifico. Eso permite que cada equipo opere solo dentro de su proyecto.

Objetivo

  • Que una persona trabaje solo en los recursos de su proyecto.
  • Que no pueda modificar ni afectar recursos de otros equipos.
  • Que la visibilidad y la capacidad operativa no sean generales para toda la organizacion.

Como se logra

  • Separando por subscription cuando se necesita aislamiento fuerte.
  • Separando por resource group cuando varios proyectos comparten suscripcion.
  • Asignando roles a grupos y no a usuarios sueltos.
Escenario Scope recomendado Rol tipico Resultado esperado
Equipo de un proyecto que administra solo sus recursos Resource group del proyecto Contributor Puede operar su solucion sin tocar otros proyectos.
Usuario que solo debe revisar configuraciones Subscription o resource group segun necesidad Reader Puede ver, pero no cambiar.
Equipo central que tambien asigna accesos Subscription o management group Owner o User Access Administrator Puede administrar recursos y delegar permisos.
Proyecto con alto aislamiento de costos, cuotas y gobierno Subscription dedicada Contributor o Reader segun funcion El proyecto queda aislado del resto a nivel operativo y administrativo.

Ejemplo A

Grupo az-rg-proyecto-a-contributors con rol Contributor en RG-Proyecto-A. Ese equipo administra solo ese grupo de recursos.

Ejemplo B

Grupo az-sub-finanzas-readers con rol Reader en una suscripcion. Pueden auditar, pero no desplegar ni modificar.

Ejemplo C

Equipo de plataforma con Owner o User Access Administrator en la suscripcion para controlar altas, bajas y asignaciones.

Idea fuerza para presentar: "Que una persona pueda ver una suscripcion no significa que pueda usarla; que pueda usar un resource group no significa que pueda operar toda la suscripcion; y que pueda administrar recursos no significa que pueda delegar permisos".
Governance

7. Suscripciones y management groups

La jerarquia tenant - management groups - subscriptions - resource groups - resources funciona como la columna vertebral de la gobernanza. Es donde se decide el alcance real del blast radius administrativo.

Patron recomendado

  • Separar suscripciones de plataforma y de workloads.
  • Usar management groups para heredar politicas y RBAC.
  • Dividir por produccion, no produccion, ownership y costo.

Gobierno complementario

  • Azure Policy define que puede desplegarse y bajo que condiciones.
  • Azure RBAC define quien puede operar sobre cada alcance.
  • Template Specs y Deployment Stacks reemplazan el enfoque a largo plazo de Blueprints.
Root management group Platform Landing zones Prod / NonProd Resource groups
Apps

8. Aplicaciones y permisos

Application object y service principal

El registro de la app define la aplicacion a nivel global. El service principal es su instancia operativa dentro del tenant y concentra permisos efectivos y grants.

Credenciales

Client secrets, certificados y credenciales federadas deben tener rotacion, expiracion y estandares. Cuando sea posible, conviene usar managed identities.

Tipo de permiso Contexto Uso tipico
Delegated Con usuario Web, SPA, movil
Application Sin usuario Daemons, backend, automatizacion
Una app con permisos app-only y admin consent excesivo puede convertirse en uno de los vectores de mayor impacto dentro del tenant.
Privilegios

9. PIM, Conditional Access y cuentas de emergencia

PIM

Reduce privilegios permanentes y habilita asignaciones elegibles y time-bound.

Conditional Access

Motor Zero Trust para exigir MFA, dispositivo compliant, ubicacion valida o bloquear riesgos.

Break glass

Cuentas cloud-only de emergencia para incidentes criticos, estrictamente controladas y monitoreadas.

PIM para roles de Entra y grupos se gestiona por Microsoft Graph; PIM para Azure resources se gestiona por Azure Resource Manager.

10. Auditoria y monitoreo

Entra sign-in logs

Separan actividad interactiva, no interactiva, service principals y managed identities.

Audit logs

Capturan cambios en usuarios, grupos, apps, licencias y configuracion de identidad.

Azure Activity Log

Bitacora de cambios en el plano de control de la suscripcion.

La practica recomendada es centralizar logs hacia Log Analytics y, cuando aplique, Sentinel, para correlacion de eventos, alertas y respuesta a incidentes.

Caso practico

11. Casos practicos para entender roles, permisos y jerarquia

Estos ejemplos ayudan a explicar por que en Azure no basta con tener un rol administrativo. Tambien importan el plano de acceso, el scope y la capa de gobernanza aplicada.

Caso 1. Key Vault: podia crear el recurso, pero no los secretos

Este ejemplo explica la diferencia entre roles, permisos, scope y los planos de acceso en Azure.

Escenario

  • Podia crear recursos en Azure.
  • Podia administrar el Key Vault como recurso.
  • No podia generar secretos dentro del Key Vault.
  • No podia dar acceso a una managed identity para consumir el Key Vault.

Interpretacion correcta

  • Tener permisos sobre el recurso no significa tener permisos sobre los datos del recurso.
  • Crear secretos pertenece al data plane.
  • Asignar acceso a otra identidad requiere permisos adicionales de administracion de acceso.
Accion Plano Que permiso se necesita Resultado del caso
Crear el Key Vault Control plane Rol de administracion del recurso, como Contributor o Key Vault Contributor Si podia hacerlo.
Crear un secreto Data plane Permisos especificos sobre secretos, por ejemplo con Azure RBAC en el vault No podia hacerlo.
Asignar permisos a la managed identity Administracion de acceso Capacidad para crear role assignments o access policies No podia hacerlo.

Administrador del recurso no es lo mismo que usuario autorizado del secreto.

Key Vault Administrator puede operar datos del vault, pero no necesariamente puede delegar acceso a otros principals.

La managed identity necesita una asignacion explicita para consumir el Key Vault; ese acceso no aparece en automatico.

Caso 2. Maquinas virtuales: podia abrir puertos, pero los cambios se revertian

Este caso explica la diferencia entre tener permiso para ejecutar un cambio y tener permiso para mantener ese cambio cuando existe una politica de gobernanza superior.

Escenario

  • Se creaban redes virtuales para asignar IP y habilitar conectividad hacia maquinas virtuales.
  • Yo podia modificar reglas de puertos para permitir conexiones.
  • Los cambios se revertian automaticamente despues.

Interpretacion correcta

  • Mi rol probablemente si permitia hacer el cambio inicial.
  • La reversion apunta a una capa de Azure Policy o remediacion automatica.
  • Esa gobernanza suele heredarse desde management group, subscription o resource group.
Comportamiento observado Lectura tecnica Capa mas probable
El cambio falla inmediatamente La solicitud fue bloqueada al evaluarse. Falta de permisos o Azure Policy con efecto deny.
El cambio aplica, pero luego vuelve al estado anterior Existe una accion correctiva posterior. Azure Policy con modify, deployIfNotExists, tarea de remediacion o automatizacion externa.
Puedo cambiar, pero no conservar la configuracion RBAC me deja operar, pero la organizacion define el estado permitido. Gobernanza heredada por scope.

Azure RBAC responde a "que puedo intentar hacer".

Azure Policy responde a "que configuracion puede existir o permanecer".

La jerarquia relevante en este caso suele ser management group -> subscription -> resource group -> recurso.

Implementacion

12. Riesgos comunes y checklist

Riesgos frecuentes

  • Asignar Owner o Contributor con scopes demasiado amplios.
  • Mantener roles legacy o classic administrators aun activos.
  • Dar admin consent a apps con permisos application demasiado extensos.
  • No rotar secretos o certificados de apps.
  • No centralizar logs de identidad y recursos.

Checklist sugerido

  • Documentar tenant y relacion de confianza con suscripciones.
  • Definir baseline de Conditional Access y validar licencias P1/P2.
  • Implementar PIM (Privileged Identity Management) priorizando eligible sobre permanent active.
  • Crear al menos dos cuentas break glass cloud-only.
  • Separar platform y landing zones con management groups.
  • Aplicar baseline de Azure Policy y exportar logs a Log Analytics.
El error mas costoso suele ser conceptual: confundir control de identidad con control de recursos. Si el diseno arranca mal en esa separacion, todo lo demas se vuelve dificil de auditar.
