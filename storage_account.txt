================================================================================
RESUMEN INTEGRAL: ARQUITECTURA, GESTIÓN Y COSTOS DE AZURE STORAGE ACCOUNTS
================================================================================

Este documento consolida los puntos críticos para el diseño, implementación y
operación profesional de cuentas de almacenamiento en Azure, estructurado desde
los fundamentos básicos hasta las consideraciones de nivel empresarial.

--------------------------------------------------------------------------------
1. PARADIGMA FUNDAMENTAL Y MODELO DE COSTOS
--------------------------------------------------------------------------------

Azure Storage funciona bajo un modelo de PAGO POR USO (Pay-as-you-go). No existe
una tarifa plana. La factura mensual es MULTIDIMENSIONAL y se calcula sumando
cinco factores principales:

1. VOLUMEN DE CAPACIDAD: Cantidad de Gigabytes (GB) almacenados físicamente.
2. TRANSACCIONES Y OPERACIONES: Cantidad de veces que se lee, escribe o lista
   la información (cobrado por bloques de 10,000 operaciones).
3. NIVEL DE ACCESO (TIER): Determina el equilibrio entre costo de
   almacenamiento y costo de acceso (ver Sección 3).
4. REDUNDANCIA: Cantidad de copias de seguridad físicas de los datos.
5. TRANSFERENCIA DE RED (EGRESS): Cobro por descargar datos o moverlos fuera de
   la región de Azure (la subida o Ingress es gratuita).

Herramienta Crítica: Utilizar siempre la 'Azure Pricing Calculator' para simular
estos costos antes de la implementación.

--------------------------------------------------------------------------------
2. SERVICIOS CENTRALES Y OPERACIONES (FUNDAMENTOS)
--------------------------------------------------------------------------------

La cuenta de almacenamiento ofrece cuatro servicios principales:

* BLOBS (OBJETOS BINARIOS): Almacenamiento masivo para datos no estructurados
    (imágenes, videos, copias de seguridad). Es el servicio más común para
    integración con AKS o Container Apps.
* FILES (RECURSOS COMPARTIDOS SMB/NFS): Almacenamiento compartido totalmente
    gestionado. Permite montar discos en múltiples VMs o Pods simultáneamente,
    ideal para migración de aplicaciones "legacy".
* QUEUES (COLAS DE MENSAJES): Sistema de mensajería asíncrona para desacoplar
    componentes de la aplicación y gestionar picos de carga.
* TABLES (NOSQL ESTRUCTURADO): Almacén de claves/valores NoSQL para acceso
    rápido a grandes volúmenes de datos estructurados, sin necesidad de un
    modelo relacional.

Capacidad Adicional: La cuenta de almacenamiento puede configurarse para ofrecer
'Hosting Web Estático', permitiendo alojar sitios front-end directamente desde un
contenedor de blobs.

--------------------------------------------------------------------------------
3. NIVELES DE ACCESO (ACCESS TIERS) Y EL "TRADE-OFF" HARDWARE
--------------------------------------------------------------------------------

El nivel de acceso afecta el costo porque refleja directamente la inversión en
hardware y energía que Microsoft realiza para garantizar el rendimiento. Existe un
equilibrio (trade-off) fundamental:

* Mayor accesibilidad y rendimiento = Mayor costo de almacenamiento, Menor costo
    de uso.
* Menor accesibilidad o uso poco frecuente = Menor costo de almacenamiento,
    Mayor costo de uso.

Desglose de los Niveles:

A. HOT (FRECUENTE)
   * Hardware: Almacenamiento online en discos de muy alto rendimiento (SSD o
       magnéticos avanzados), siempre encendidos y optimizados para tráfico masivo.
   * Costo Almacenar: Más alto.
   * Costo Operar (Leer/Escribir): Más bajo.
   * Disponibilidad: Instantánea.
   * Uso Ideal: Datos transaccionales, logs en tiempo real, archivos de acceso
       recurrente.

B. COOL (ESPORÁDICO)
   * Hardware: Almacenamiento online en discos de menor costo y rendimiento.
       Microsoft ahorra en infraestructura, permitiendo un menor costo de
       almacenamiento.
   * Costo Almacenar: Bajo (aprox. la mitad que Hot).
   * Costo Operar (Leer/Escribir): Medio (más caro que Hot).
   * Disponibilidad: Instantánea (o latencia muy baja).
   * Uso Ideal: Copias de seguridad mensuales, datos de telemetría antiguos,
       archivos de acceso raro pero inmediato.
   * Penalización: Retención mínima de 30 días.

C. ARCHIVE (ARCHIVO) - EL PROCESO DE REHIDRATACIÓN
   * Hardware: Almacenamiento OFFLINE en hardware de ultra-baja velocidad y alta
       densidad (generalmente cintas), a menudo en reposo profundo. Esto reduce
       el costo de energía casi a cero.
   * Costo Almacenar: Minúsculo (regalado).
   * Costo Operar (Leer): Extremo (incluye la penalización de rehidratación).
   * Disponibilidad: NO ES ACCESIBLE DIRECTAMENTE.
   * Proceso Interno (Rehidratación): Para leer un dato, se debe realizar una
       solicitud para cambiar su nivel a Hot o Cool. Este proceso interno mueve
       el dato desde el hardware offline al online y puede tardar de horas
       (Estándar) a minutos (Prioridad, más caro).
   * Uso Ideal: Datos de conservación legal por años, copias de seguridad de
       seguridad extrema.
   * Penalización: Retención mínima de 180 días.

4. RESILIENCIA Y PROTECCIÓN DE DATOS
--------------------------------------------------------------------------------

Planificar contra fallos y errores humanos:

* REDUNDANCIA:
    * LRS (LOCAL): 3 copias en el mismo centro de datos (económico, protege
        contra fallos de disco).
    * ZRS (ZONA): 3 copias en zonas de disponibilidad separadas. Recomendable
        para alta disponibilidad regional (AKS/Container Apps).
    * GRS (GEOGRÁFICA): 6 copias (3 locales y 3 en otra región geográfica). Protege contra desastres regionales mayores.
* PROTECCIÓN CONTRA ELIMINACIÓN:
    * SOFT DELETE (ELIMINACIÓN SUAVE): Papelera de reciclaje para blobs/files.
    * BLOB VERSIONING: Historial automático de versiones anteriores de archivos.
    * ALMACENAMIENTO INMUTABLE (WORM): Bloquea contenedores para que no puedan
        ser modificados ni eliminados por un periodo definido legalmente.
* AUTOMATIZACIÓN: 'Lifecycle Management' permite crear reglas para mover
    automáticamente datos entre tiers (ej: Hot -> Cool -> Archive -> Delete)
    basado en la antigüedad.

--------------------------------------------------------------------------------
5. ARQUITECTURA ENTERPRISE Y SEGURIDAD (EL PILAR CRÍTICO)
--------------------------------------------------------------------------------

Integración segura de la cuenta de almacenamiento en el ecosistema corporativo:

* SEGURIDAD DE RED: Por defecto, es pública por internet. En producción, se
    DEBE restringir el acceso. La herramienta clave son los PRIVATE ENDPOINTS
    (Azure Private Link), que asignan una IP privada interna a la Storage
    Account dentro de una VNET, eliminando la exposición a internet.
* SEGURIDAD DE IDENTIDAD: Desactivar el uso de Access Keys compartidas. Utilizar
    AZURE ACTIVE DIRECTORY (AZURE AD) con RBAC para asignar roles específicos a
    identidades administradas (Managed Identities) de las aplicaciones, asegurando
    el principio de mínimo privilegio.
* GOBERNANZA: Utilizar AZURE POLICY para imponer mejores prácticas
    automáticamente (ej: prohibir cuentas con acceso público activado).
* BIG DATA ANALYTICS: Activar el 'Hierarchical Namespace' (ADLS Gen2) para
    que la cuenta de blobs se comporte como un sistema de archivos real con
    jerarquías de carpetas verdaderas, crucial para herramientas como
    Databricks o Synapse Analytics.
* REPLICACIÓN DE OBJETOS: Configurar replicación asíncrona de blobs entre
    cuentas de diferentes regiones para Recuperación ante Desastres (DR) regionales.
